Database Validation

Show how validation reduces bad data entry

OCN NI Level 2–3 — Databases

Learning Objectives

By the end of this session, you will be able to:

  • Explain why bad data costs time and trust
  • Apply field rules (required, type, range, format)
  • Compare validated vs unvalidated entry live
  • Choose sensible validation for a sample form
  • Link validation to data quality and GDPR accuracy

Garbage In, Garbage Out

If a form accepts anything, reports lie. Validation stops nonsense before it lands in the database — required fields, correct types, sensible ranges, and allowed lists.

Common Validation Rules

RuleExampleBlocks
RequiredEmail must not be blankEmpty customer records
Data typeAge must be a numberText in numeric fields
Range / lengthQty 1–100; NI number formatImpossible values
Lookup / listTown from approved listMisspellings & freestyle

Tip

Validation at entry is cheaper than cleaning later.

Demo Sequence

  1. Open a form without validation — enter blank name, letters in Age, negative price
  2. Show how those rows pollute a simple report/query
  3. Enable required, numeric, and range rules on the same form
  4. Retry the bad entries — show clear error messages
  5. Discuss: which rules belong in the form vs the database?

Test Your Knowledge!

Let's see what you've learned about database validation

Q1: Validation mainly aims to…

AMake forms prettier
BReduce incorrect or incomplete data at entry
CDelete all tables nightly
DReplace backups
Answer: B — Good rules stop bad data before it spreads into reports.

Q2: A range check on Quantity might…

AAllow any text
BReject 0 or 10,000 if the rule is 1–500
CSort the table
DCreate a JOIN
Answer: B — Ranges define sensible minimum and maximum values.

Q3: Leaving Email optional when you need contact details…

AImproves security always
BRisks incomplete records you cannot use
CIs required by SQL syntax
DAutomatically validates phone numbers
Answer: B — Required fields enforce the data you actually need.
