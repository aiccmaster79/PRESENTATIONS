#31·Level 5·Cyber Security

Vulnerable Web App Walkthrough

Learn common web risks safely in an isolated lab

OCN NI Level 5 — Cyber Security

Learning Objectives

By the end of this session, you will be able to:

  • Explain why deliberately vulnerable apps are used for training
  • Identify common OWASP-style web risks (XSS, SQLi, weak auth)
  • Demonstrate safe exploration of DVWA or OWASP Juice Shop
  • Recommend practical defences for each risk shown
  • Apply legal and lab isolation rules for offensive demos

Why a Vulnerable App?

Real websites must never be attacked. Labs like DVWA and OWASP Juice Shop are built to be broken — so learners can see risks without harming systems or people.

Classroom rule

Only use the lab VM/network provided. No scanning of college, home, or public sites.

Risks You Will See

SQL Injection

Untrusted input reaches the database

Cross-Site Scripting

Scripts run in another user’s browser

Broken Auth

Weak logins and session handling

Sensitive Data

Secrets left in clear text

Insecure Upload

Malicious files accepted

Misconfiguration

Debug modes and default settings

Lab vs Real World

Allowed in the lab

  • DVWA / Juice Shop on isolated VMs
  • Documented tutor permission
  • Screenshots for evidence
  • Discuss fixes and controls

Never do this

  • Attack live websites
  • Scan classmates’ phones/PCs
  • Share exploit payloads outside class
  • Skip written authorisation

Demo Sequence

  1. Confirm the lab is isolated (no route to the internet / college LAN)
  2. Open DVWA or Juice Shop and set difficulty / scoreboard as needed
  3. Walk through one injection or XSS challenge end-to-end
  4. Show the vulnerable code pattern (or request) that made it possible
  5. Map each finding to a defence: validation, parameterised queries, CSP, MFA
  6. Learners complete one guided challenge and note the control that would stop it

Test Your Knowledge!

Let's see what you've learned about vulnerable web applications

Q1: Why use DVWA or Juice Shop instead of a real website?

AReal sites are safer to attack
BThey are designed for safe training with known flaws
CThey have no security relevance
DThey replace the need for permission
Answer: B — Deliberately vulnerable apps let you practise without harming real users or systems.

Q2: Which control best stops classic SQL injection?

ABigger monitors
BParameterised queries / prepared statements
CTurning off HTTPS
DSharing admin passwords
Answer: B — Keep untrusted input out of the query structure.

Q3: Before any lab exploit demo you must…

APost results publicly
BConfirm isolation and authorised scope
CDisable all logging
DUse the college production server
Answer: B — Authorisation and isolation are non-negotiable.
