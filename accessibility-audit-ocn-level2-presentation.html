Accessibility Audit

Check a webpage for accessibility issues

OCN NI Level 2–3 — Website development

Learning Objectives

By the end of this session, you will be able to:

  • Explain why accessibility matters (users, law, quality)
  • Run a basic audit using browser tools and a checklist
  • Identify issues: contrast, alt text, headings, focus, labels
  • Fix at least two issues in the demo site files
  • Retest to show improvement

Access Is a Feature

Accessibility (a11y) means people with diverse abilities can perceive, operate and understand your site — keyboard users, screen reader users, low vision, and more.

Mini Audit Checklist

CheckPass looks like…Fail looks like…
Alt textMeaningful descriptionMissing or empty on content images
ContrastClear textGrey on grey / orange on orange
HeadingsOne h1, ordered levelsSkipped levels / all bold paragraphs
Forms<label for> linkedPlaceholder-only fields
KeyboardReach all controlsFocus trapped / invisible

Demo Sequence

  1. Open a demo page seeded with accessibility issues
  2. Walk the checklist manually (headings, alt, labels)
  3. Use DevTools Accessibility / contrast helpers or Lighthouse Accessibility score as a guide
  4. Tab through the page with the keyboard — note focus problems
  5. Fix two issues live in VS Code (alt text + label or contrast)
  6. Reload and re-audit — show the before/after

Q1: Alt text is mainly for…

ADecorating CSS
BDescribing meaningful images when they can’t be seen
CSpeeding up Wi‑Fi
DEncrypting passwords
Answer: B — Screen readers and text-only contexts use alt text.

Q2: Poor colour contrast hurts…

AOnly printers
BUsers with low vision (and many others)
COnly servers
DOnly Git commits
Answer: B — Text must be readable against its background.

Q3: A labelled form input helps…

AScreen reader and all users understand the field
BThe CPU run cooler
CMacros record faster
DVLANs route better
Answer: A — Labels programmatically describe inputs.
