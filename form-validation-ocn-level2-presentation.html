Form Validation

HTML5 vs JavaScript validation

OCN NI Level 2–3 — Website development

Learning Objectives

By the end of this session, you will be able to:

  • Build a simple form with HTML5 validation attributes
  • Compare built-in browser validation with custom JavaScript checks
  • Use required, type, min/max, pattern (overview)
  • Show a JS example that validates on submit with custom messages
  • Explain why you still validate on the server in real apps

Two Layers

HTML5

  • required, type="email"
  • min / max / pattern
  • Fast to add; browser messages
  • Limited custom UX

JavaScript

  • Custom rules and messages
  • Cross-field checks (password confirm)
  • Can style errors your way
  • Must not be the only defence

Side-by-Side Sketch

<!-- HTML5 -->
<input type="email" name="email" required>

<!-- JS (idea) -->
form.addEventListener("submit", (e) => {
  if (!email.value.includes("@")) {
    e.preventDefault();
    error.textContent = "Enter a valid email.";
  }

Reality check

Client-side checks improve UX. Server-side validation is still required — browsers and JS can be bypassed.

Demo Sequence

  1. Open a demo form with no validation — submit empty/bad data
  2. Add HTML5 required + type=email + minlength; show browser bubbles
  3. Disable HTML5 with novalidate and validate in JavaScript instead
  4. Show a custom error message region for clearer UX
  5. Compare: speed of HTML5 vs flexibility of JS
  6. Discuss server-side validation as the non-negotiable backstop

When to Use What

HTML5 first

Great for required, email, number ranges

Add JS when…

Custom messages, multi-field rules, live feedback

Always server

Never trust the client alone

Accessible errors

Link messages to fields; don’t rely on colour only
